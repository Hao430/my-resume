# Pentagon Workflow 总控中枢

**版本**: 0.2 (智能入口增强版)
**最后更新**: 2026-01-27

## 1. 元规则
1.  本工作流是一个基于**状态机**的自动化协作系统，严格遵循 PDCA 循环。
2.  系统的核心目标是实现**完全自动化**。人类用户的唯一常规职责是：在智能体提出明确的是/否问题时，回答“是/Y”或“否/N”。**当 `goal.md` 定稿后，若用户未通过编辑 `correct.md` 提出修正要求，系统将持续自动执行后续周期，直至目标达成。**
3.  智能体（Agent）在下方“路由表”定义的触发条件下自动运行，驱动循环。
4.  所有操作均围绕五个核心文件进行，必须维护其状态的一致性与可追溯性。
5.  `review.md` 作为**活体知识库**，采用倒序排列，包含全部历史循环记录，并具备自我导航能力。

## 2. 状态-智能体路由表 (自然问答链条)
| 触发条件（当文件状态变为） | 自动触发的智能体 | 预期产出与动作 |
| :--- | :--- | :--- |
| `goal.md` ➜ **`定稿`** | `pentagon-roadmap-maker` | 生成 `roadmap.md` 并自动设为"定稿"状态，无需用户确认。 |
| `roadmap.md` ➜ **`定稿`** | `pentagon-todo-maker` | 生成 `todo.md`，状态设为 **`未完成`**。 |
| `todo.md` ➜ **`未完成`** | `pentagon-todo-executor` | 更新 `todo.md`，记录任务执行情况，将状态设为 **`已完成`**。 |
| `todo.md` ➜ **`已完成`** | `pentagon-review-maker` | 生成 `review.md` 完成情况记录，更新 `cost.md`（如果存在），状态设为 **`草稿`**。 |
| `review.md` ➜ **`草稿`** | `pentagon-review-maker` | 分析完成情况，生成复盘结论，无需用户确认。 |
| `review.md` ➜ **`定稿`** | `pentagon-cycle-coordinator` | 检查成本限制（如果 `cost.md` 存在），评估是否需要修改 `goal.md`，如需要则自动修改并设置为 **`定稿`**，应用并归档 `correct.md` 内容，然后清空 `correct.md`，提交变更，最后调用 `pentagon-roadmap-maker` 继续执行。 |

## 3. 智能入口路由表 (ptg命令专用)
| 用户输入状态 | 自动触发的智能体 | 预期产出与动作 |
| :--- | :--- | :--- |
| 用户执行 `/ptg [任务描述]` | `/ptg` 命令 | 检查是否存在有效的 `goal.md`，决定下一步行动 |
| 存在有效的 `goal.md` 且状态为 **`定稿`** | `/ptg` 命令 | 检查并应用 `correct.md`（如果存在），然后触发 `pentagon-roadmap-maker` |
| 存在有效的 `goal.md` 且状态为 **`草稿`** | `/ptg` 命令 | 展示 `goal.md` 内容，要求用户确认 |
| 不存在或无效的 `goal.md` | `pentagon-clarifier` | 评估任务描述清晰度，输出评估结果和行动建议 |
| `pentagon-clarifier` 评估为 **A级** (≥80分) | `/ptg` 命令 | 直接生成 `goal.md`，状态设为 **`草稿`**，并启动后续工作流 |
| `pentagon-clarifier` 评估为 **B/C级** (60-79分) | `pentagon-clarifier` | 提出1-3个关键澄清问题，等待用户回答 |
| `pentagon-clarifier` 评估为 **D/F级** (<60分) | `pentagon-clarifier` | 引导式对话逐步构建任务，或提供任务设计指南 |

## 4. 文件状态契约与自然问答流转
每个文件的生命周期遵循以下状态顺序，**`→` 表示自动流转**：

### `goal.md` (目标)
- **`草稿`** → **`定稿`**  → **`达成`**
- *说明：由初始化命令（如`/ptg`）创建为`草稿`。`/ptg` 命令会使用 `pentagon-clarifier` 评估任务清晰度，如果评估为 A 级则直接生成 `goal.md`（草稿），如果评估为 B/C 级则提出澄清问题等待用户回答，如果评估为 D/F 级则引导用户重填。当 `goal.md` 状态变为"定稿"时，自动触发 `pentagon-roadmap-maker` 生成路线图。当全部目标达成后，智能体自动将状态更新为 **`达成`**，停止工作循环。*

### `roadmap.md` (路线图)
- **`定稿`**
- *说明：由`pentagon-roadmap-maker`生成后自动设为`定稿`，无需用户确认，自动触发任务分解。*

### `todo.md` (任务清单)
- **`未完成`** → **`已完成`**
- *说明：由`pentagon-todo-maker`自动生成。任务完结时由`pentagon-review-maker`自动标记为`已完成`。*

### `review.md` (智能复盘知识库)
- **`草稿`** → **`定稿`**
- *说明：由`pentagon-review-maker`自动生成完成情况记录和执行回顾，更新 `cost.md`（如果存在），生成 `review.md`，状态为 **`定稿`**。之后系统自动检查成本限制、执行决策、应用 `correct.md` 内容并清空 `correct.md`，无需用户确认即可推进下一周期。*

### `correct.md` (系统修正文件)
- **`存在`** → **`已归档并清空`**
- *说明：用户可随时写入修正意见。当 `review.md` 定稿时，`pentagon-cycle-coordinator` 会读取并应用修正，归档到 `review.md` 后清空。文件不存在或为空表示无需修正。*

### `cost.md` (成本管理文件，可选)
- **`存在`** → **`持续更新`**
- *说明：用户可创建此文件设置成本限制。`pentagon-review-maker` 每次循环后更新 tokens 和费用信息。`pentagon-cycle-coordinator` 检查是否超出限制，超出则停止工作流。文件不存在则不进行成本管理。*

## 5. 智能入口决策逻辑
用户通过 `/ptg` 命令启动任务时，系统自动评估输入清晰度：

| 评估等级 | 分数范围 | 自动决策 | 用户交互 |
| :--- | :--- | :--- | :--- |
| **A级** | 80-100分 | 直接生成 `goal.md` 并启动工作流 | 无额外交互，直接进入自动化流程 |
| **B级** | 70-79分 | 提出1-2个关键澄清问题 | 用户回答后重新评估或直接生成 |
| **C级** | 60-69分 | 提出2-3个结构化澄清问题 | 用户填写缺失信息后生成 |
| **D级** | 40-59分 | 提供标准模板引导重填 | 引导式对话，逐步明确需求 |
| **F级** | 0-39分 | 拒绝执行，提供任务设计指南 | 教育用户如何描述明确任务 |

## 6. pentagon-clarifier 评估维度与权重
```
1. 目标明确性 (30%)
   - 是否有明确的产出物？
   - 是否有可衡量的成功标准？
   - 是否有具体的时间/质量约束？

2. 上下文完整性 (25%)
   - 是否包含必要背景信息？
   - 是否有明确的输入/输出定义？
   - 是否有相关约束条件说明？

3. 执行可行性 (25%)
   - 任务是否可分解为具体步骤？
   - 是否有明确的完成标准？
   - 是否有边界条件定义？

4. 术语精确性 (20%)
   - 是否避免模糊词汇？
   - 是否使用领域专业术语？
   - 是否有明确的量化指标？
```

| 交互场景 | 智能体提问 | 用户回答 | 触发的智能逻辑 |
| :--- | :--- | :--- | :--- |

## 8. 文件结构优化 - review.md 活体知识库规范

为了解决上下文长度问题并提升知识管理效率，`review.md` 必须遵循以下结构：

```markdown

**最新摘要**: [此处由AI自动更新，用1-2句话概括最近3个循环的核心结论]
**当前待决问题**: [来自最近一次`review`或`correct`的关键待办项]

---

## 📋 历史循环快速索引 (由系统自动维护)
| 循环标识 | 时间 | 核心决策 | 详情章节 |
| :--- | :--- | :--- | :--- |
| `cycle-15` | 2025-01-26 | 批准了V2架构；决定增加性能测试。 | 跳转至 [#cycle-15](#cycle-15) |
| `cycle-14` | 2025-01-19 | 调整了需求优先级。 | 跳转至 [#cycle-14](#cycle-14) |
| ... | ... | ... | ... |

---

## 🔍 详细复盘记录 (以下为完整历史，按时间倒序排列)

### #cycle-15 (2025-01-26)
**状态**: 已归档
**应用的系统修正**:
- [来自correct.md的内容，在应用后已清空]
**复盘分析与决策**:
- [本次循环的完整分析内容...]

```

### 智能体读取规则
| 智能体 | 需要读取 `review.md` 的哪部分？ | 指令设计关键 |
| :--- | :--- | :--- |
| **`pentagon-cycle-coordinator`** | 1. **"最新摘要"**和**"当前待决问题"**。<br>2. **最近1-2个**详细循环记录。 | "在协调循环和应用修正前，请读取`review.md`顶部的摘要和最近一次完整复盘，以理解项目最新状态。" |

| **`pentagon-roadmap-maker`** | **"当前待决问题"**和**最近1次复盘**中的相关决策。 | "制定路线图时，请关注`review.md`顶部指出的待决问题，确保你的计划能解决它。" |

## 9. correct.md 处理流程

`correct.md` 作为临时修正缓冲区，其处理流程如下：

1. **写入**：人类用户可以在任何时刻向 `correct.md` 写入修正意见
2. **读取**：
   - `/ptg` 命令在执行前会检查并应用 `correct.md`（如果 `goal.md` 状态为"定稿"）
   - `pentagon-cycle-coordinator` 在协调循环时也会读取 `correct.md` 内容（如果文件存在）
3. **应用**：将修正内容应用到 `goal.md` 的修改中
4. **归档**：
   - **如果 `correct.md` 存在且非空**：将 `correct.md` 内容追加到 `review.md` 的当前循环章节中
   - **如果 `correct.md` 不存在或为空**：在 `review.md` 中记录"用户未提供修正意见，无需修正"
5. **清空**：
   - **如果 `correct.md` 存在且非空**：清空 `correct.md` 文件内容，准备下一轮使用
   - **如果 `correct.md` 不存在或为空**：无需清空操作

此机制确保了人类智慧被及时应用、永久记录，并避免了知识丢失。同时，允许 `correct.md` 不存在或为空，表示用户认为当前迭代无需修正。

## 10. cost.md 成本管理流程

`cost.md` 作为可选的成本管理文件，其处理流程如下：

1. **创建**：用户可选择创建 `cost.md` 文件并设置成本限制
2. **配置**：用户在 `cost.md` 中设置：
   - tokens 限制
   - 百万 tokens 费用
   - 费用限制
   - 币种
3. **更新**：
   - `pentagon-review-maker` 每次循环后统计本次使用的 tokens
   - 计算本次费用并更新累计费用
   - 将本次记录添加到历史记录表格
4. **检查**：
   - `pentagon-cycle-coordinator` 在协调循环时检查 `cost.md`（如果存在）
   - 如果累计 tokens ≥ tokens 限制，停止工作流
   - 如果累计费用 ≥ 费用限制，停止工作流
5. **状态**：
   - **正常**：未超出任何限制
   - **警告**：接近限制（可选，可设置预警阈值）
   - **已停止**：超出限制，工作流已暂停

此机制确保用户可以控制 AI 使用成本，避免意外超支。`cost.md` 为可选功能，不创建该文件则不进行成本管理。
