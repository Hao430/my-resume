## 二、数据库设计

### 表结构

```sql
-- 简历数据（JSONB 存储各段，避免频繁改表结构）
CREATE TABLE resume (
  id          SERIAL PRIMARY KEY,
  section     TEXT NOT NULL UNIQUE,  -- 'personal_info' | 'summary' | 'skills' | 'projects' | 'experience' | 'honors'
  data        JSONB NOT NULL,
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 博客文章元数据 + 正文
CREATE TABLE blog_posts (
  id           SERIAL PRIMARY KEY,
  slug         TEXT NOT NULL UNIQUE,
  title        TEXT NOT NULL,
  description  TEXT NOT NULL DEFAULT '',
  date         DATE NOT NULL,
  tags         TEXT[] NOT NULL DEFAULT '{}',
  external_url TEXT,
  content_html TEXT,                 -- 文章正文 HTML
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 演示文稿
CREATE TABLE slides (
  id           SERIAL PRIMARY KEY,
  slug         TEXT NOT NULL UNIQUE,
  title        TEXT NOT NULL,
  description  TEXT NOT NULL DEFAULT '',
  page_count   INTEGER NOT NULL DEFAULT 0,
  date         DATE NOT NULL,
  content_html TEXT,                 -- 演示 HTML 内容
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 每日早参
CREATE TABLE daily_briefs (
  id            SERIAL PRIMARY KEY,
  date          DATE NOT NULL UNIQUE,
  title         TEXT NOT NULL,
  content_html  TEXT NOT NULL,
  display_date  TEXT,                -- 如 "4月13日 周一"
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 索引
CREATE INDEX idx_blog_posts_date ON blog_posts(date DESC);
CREATE INDEX idx_daily_briefs_date ON daily_briefs(date DESC);
CREATE INDEX idx_slides_date ON slides(date DESC);
```

### 设计说明

- 简历用 JSONB 而非多表：数据量极小（一条简历），JSONB 避免了 6 张表的 join 开销，修改结构无需迁移
- 文章 HTML 直接存 DB：省去文件系统读写，部署时少一个 concern
- `public/*.html` 仍保留：过渡期兼容旧引用，后期可完全迁移到 DB

## 三、API 接口

### 基础约定

- 基准路径：`/api`
- Content-Type：`application/json`
- 错误响应格式：`{ "error": string, "details?": unknown }`

### 端点列表

| 方法 | 路径 | 说明 | 数据来源 |
|------|------|------|---------|
| GET | `/api/resume` | 全部简历数据（所有 section 合并为一个对象） | resume 表 |
| GET | `/api/resume/:section` | 单段简历数据 (e.g. `skills`, `projects`) | resume 表 |
| GET | `/api/posts` | 博客列表，支持 `?tag=xxx` 筛选 | blog_posts 表 |
| GET | `/api/posts/:slug` | 单篇文章详情（含 content_html） | blog_posts 表 |
| POST | `/api/posts` | 新增文章 | blog_posts 表 |
| PUT | `/api/posts/:slug` | 更新文章 | blog_posts 表 |
| DELETE | `/api/posts/:slug` | 删除文章 | blog_posts 表 |
| GET | `/api/briefs` | 早参列表（按日期倒序） | daily_briefs 表 |
| GET | `/api/briefs/:date` | 某日早参详情 | daily_briefs 表 |
| GET | `/api/slides` | 演示列表 | slides 表 |
| GET | `/api/slides/:slug` | 单个演示详情 | slides 表 |

## 四、前端改动

### 4.1 新增 API 请求层 `client/src/api/index.ts`

```typescript
const API_BASE = import.meta.env.VITE_API_URL || '/api'

async function get<T>(path: string): Promise<T> {
  const res = await fetch(`${API_BASE}${path}`)
  if (!res.ok) {
    throw new Error(`API error: ${res.status} ${res.statusText}`)
  }
  return res.json()
}

// 只暴露前端需要的读接口，写接口可后续按需添加
export const api = {
  getResume:        ()                  => get<ResumeData>('/resume'),
  getResumeSection: (section: string)   => get<unknown>(`/resume/${section}`),
  getPosts:         (tag?: string)      => get<BlogPost[]>(`/posts${tag ? `?tag=${tag}` : ''}`),
  getPost:          (slug: string)      => get<BlogPost>(`/posts/${slug}`),
  getBriefs:        ()                  => get<DailyBrief[]>('/briefs'),
  getSlides:        ()                  => get<Slide[]>('/slides'),
}
```

### 4.2 改造 Pinia Store

**blog.ts** — 从硬编码改为 API 获取：

```typescript
export const useBlogStore = defineStore('blog', () => {
  const posts = ref<BlogPost[]>([])
  const loading = ref(false)
  const error = ref<string | null>(null)

  async function fetchPosts() {
    loading.value = true
    error.value = null
    try {
      posts.value = await api.getPosts()
    } catch (e) {
      error.value = e instanceof Error ? e.message : '加载失败'
    } finally {
      loading.value = false
    }
  }

  function getPostBySlug(slug: string): BlogPost | undefined {
    return posts.value.find(p => p.slug === slug)
  }

  return { posts, loading, error, fetchPosts, getPostBySlug }
})
```

**slides.ts** — 同上，去掉硬编码，改为 `fetchSlides()`。

### 4.3 修改页面

| 页面 | 改动内容 |
|------|---------|
| `HomePage.vue` | `import { resumeData }` → `store.resume` 从 Pinia 或 API 获取 |
| `AboutPage.vue` | 同上 |
| `BlogPage.vue` | store 接口不变，只改 store 内部实现 |
| `DailyBriefPage.vue` | `import { dailyBriefs }` → 从 API 获取 |
| `BlogPostPage.vue` | 从 `getPostBySlug` 改为调 API `/posts/:slug` |
| `BlogReaderPage.vue` | 不变（仍是 iframe 加载 HTML） |

### 4.4 应用初始化

在 `App.vue` 或 `main.ts` 的 `onMounted` 中触发首次数据加载：

```typescript

app.use(createPinia())
app.use(router)

// 启动时加载全局数据

app.mount('#app')
```

## 六、开发与部署

### 本地开发

```yaml
# docker-compose.yml
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: resume
      POSTGRES_USER: resume
      POSTGRES_PASSWORD: secret
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U resume"]
      interval: 5s
      timeout: 5s
      retries: 5

volumes:
  pgdata:
```

启动流程：

```bash
# 1. 启动 PostgreSQL
docker compose up -d postgres

# 2. 初始化数据库
cd server && npx tsx src/db/schema.sql

# 3. 导入种子数据
npx tsx src/seed/index.ts

# 4. 启动后端
npx tsx watch src/index.ts          # http://localhost:3000

# 5. 启动前端（另一终端）
cd client && npm run dev              # http://localhost:5173
```

### 线上部署

选项 A — **前后端分离部署**（推荐）：

| 组件 | 部署方式 |
|------|---------|
| 前端 (`client/`) | Vite build → 静态文件 → Nginx / CDN |
| 后端 (`server/`) | Express 服务 → Railway / Fly.io / 云服务器 |
| 数据库 | PostgreSQL → 云数据库（Railway 自带 / Supabase / Neon） |

选项 B — **单机部署**：

```
Nginx → /api/* → localhost:3000 (Express)
      → /*     → /path/to/client/dist (静态文件)
```

前端构建产物与后端同机部署，Nginx 反代 API。

### 环境变量

```
# server/.env
DATABASE_URL=postgres://resume:secret@localhost:5432/resume
PORT=3000

# client/.env（可选，默认 Vite proxy）
VITE_API_URL=/api
```

## 附录：为什么要去掉 MongoDB

上一轮讨论的双数据库（PgSQL + MongoDB）在这个项目中不适用：

- **你的数据全是结构化的**：简历、博客、早参都有固定的字段结构，没有非结构化文档或深层嵌套
- **没有高吞吐写入场景**：个人站几乎没有并发写，MongoDB 的写入优势用不上
- **PostgreSQL JSONB 可替代**：简历的各段数据天然适合 JSONB，既保留灵活性又省去一套运维

保持单数据库，降低心智负担和运维成本。
